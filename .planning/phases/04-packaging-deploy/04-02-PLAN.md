Create docker-compose.yml at the repo root with the following exact content:

```yaml
version: "3.8"

services:
  akc:
    build: .
    ports:
      - "8080:8080"
    environment:
      LLM_MODEL: ${LLM_MODEL:-gpt-4o-mini}
      LLM_BASE_URL: ${LLM_BASE_URL:-http://localhost:8000}
      LLM_API_KEY: ${LLM_API_KEY:-test-key}
      MEMORY_ID: ${MEMORY_ID:-test-memory}
      AKC_KB_DIR: /app/data
    volumes:
      - ./kb_data:/app/data
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 5s
      timeout: 2s
      retries: 3
      start_period: 10s
```

Exact values required:
- build: . (relative to repo root where Dockerfile lives)
- port mapping: "8080:8080"
- AKC_KB_DIR: /app/data (hardcoded — not an env var override)
- volume mapping: ./kb_data:/app/data (host ./kb_data/ directory maps to container /app/data)
- healthcheck test: curl -f http://localhost:8080/health
- healthcheck interval: 5s
- healthcheck timeout: 2s
- healthcheck retries: 3
- healthcheck start_period: 10s

Environment variable defaults use ${VAR:-default} syntax so developers can override via .env or command line. The default values (gpt-4o-mini, localhost:8000, test-key, test-memory) are safe placeholders that allow the container to start for structural testing without real API credentials.

Do NOT add GREENNODE_CLIENT_ID, GREENNODE_CLIENT_SECRET, or GREENNODE_AGENT_IDENTITY to the environment section — these are AgentBase-specific and not needed for local testing.
Do NOT add named volumes (volumes: section at the top level) — use an anonymous host path mount (./kb_data:/app/data) so the data directory is visible on the host for easy inspection.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| docker-compose.yml env defaults → container | Default values (test-key, test-memory) are placeholders; real credentials must be provided via .env file or shell environment — never committed to repo |
| host ./kb_data/ ↔ container /app/data | Volume mount exposes container's pattern data to host filesystem; only appropriate for local dev, not production (AgentBase manages this in prod) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-06 | Information Disclosure | docker-compose.yml env defaults | mitigate | Default values (test-key, gpt-4o-mini) are safe placeholders with no real credentials; real credentials loaded from .env file which is excluded by .gitignore and .dockerignore — never committed |
| T-04-07 | Information Disclosure | ./kb_data/ host directory | mitigate | .gitignore prevents kb_data/ from being committed; directory contains only local test patterns (no PII or secrets); host filesystem access is standard for local dev environments |
| T-04-08 | Tampering | docker-compose.yml volume mount | accept | ./kb_data:/app/data allows host to write directly to container's KB directory; this is intentional for local testing (developer can inspect and modify patterns.jsonl); not a threat in local dev context |
| T-04-09 | Denial of Service | healthcheck retries: 3 | accept | healthcheck is advisory only in local dev; if /health fails 3 times, container is marked unhealthy but continues running; not a production safety mechanism |
</threat_model>
